Spring事务@Transactional注解

Spring事务原理

spring事务依赖于spring aop，AOP的实现机制有两种：Proxy-based和Weaving-based，前者是依赖动态代理的方式达到对代理类增强的目的。后者是通过字节码增强的方式达到增强的目的。而在Spring事务中，一般默认使用的是前者，这意味着必须创建动态代理对象。Spring事务，在PROXY模式下，spring会使用jdk动态代理（基于接口的）或者cglib动态代理（继承生成子类）创建代理对象；而生成的代理对象会依赖于底层的事务能力，在spring的应用层对它进行了抽象。通过它，我们可以实现业务逻辑层的更大范围的事务。

Spring声明AOP的方式也有两种，一种是通过声明Aspect，另一种是通过声明Advisor（Spring 事务使用的方式）。无论是哪种方式，都需要表达清楚你要进行增强的逻辑 (what)和你要增强的地方(where)。即，需要告诉Spring你要增强什么逻辑，并且对哪些Bean/哪些方法增强。这里的what和where换成AOP中的概念分别就是对应Advice和Pointcut。类似地，我们大概可以理解AOP的思想，以一种面向切面的思想去对原来的方法做统一增强，我们可以实现，对目标方法的调用行为和参数记录日志，设置编码操作。AOP技术实现离不开advice(确定要增强的逻辑)，poincut（定位哪些地方要增强，可以通过表达式实现）的概念，通过pointcut我们确定了要被增强的位置，然后对advice做排序并穿插在原方法的逻辑的周围执行。Spring事务也是如此，织入了事务的处理过程，对使用了@Transactional，确定了要被增强的地方，并对原来方法做了事务增强。

事务失效场景

  1. 同一类中调用了事务方法
public class A {
    a() {
        b();
    }
    //声明事务
    @Transactional
    b() {
        sql操作
    }
}

/**
*如果这个时候直接调用a()方法，那么在b方法运行错误的时候，即使设置了异常时回滚，那么也不会回滚。原因如下类A经过Spring AOP 生成代理对象Proxy$A
*/


public class Proxy$A {
     A a = new A();
      
     a() {
         a.a();
     }
      
     b() {
        //开启事务
        startTransaction();
        a.b();
        commitTransactionAfterReturning();
     }
      
}

/**
*在代理对象Proxy$A调用方法a时，实际调用被代理对象的方法a()，即a.a()。那么在a.a()中，会调用原对象的方法b()，而不是调用代理对象Proxy$A的b()方法。所以实际上是没有到达事务拦截器的。所以会发现，即使生产了代理对象Proxy$A，但是事务失效了。
解决方法：b方法在新的service中定义。这样就会被拦截生效。
简单解释下：因为在新的service中定义b方法时。
*/

public class A {
    @Autowired
    B b;
 
 
 
    a() {
        b.b();
    }
 
}
 
 
public class B {
    //声明事务
    @Transactional
    b() {
        sql操作
    }
}

public class Proxy$B {
     B b = new B();
      
     b() {
        //开启事务
        startTransaction();
        b.b();
        commitTransactionAfterReturning();
     }
      
}
 
 
//在调用classA的a方法时，实际会生成代理对象Proxy$B，然后调用Proxy$B.b()方法 ，即会被方法拦截到，事务生效。

  1. 数据库引擎不支持，例如mysql的 MyISAM引擎

  2. 未指定RollBackOn，默认为RuntimeException及其子类，然后进行回滚操作

  3. 非公开non-public方法上，即便是（protected和default方法，private），事务不生效。final关键字也不可行，无法生成子类
